# Progression Cue Deadlock

## Status

Open architecture issue discovered during the full Vertex AI session test on July 8, 2026.

Reproduction session:

```text
3e086a8f-933d-4b1d-bc4d-24b857305876
```

This issue is independent of the earlier Gemini free-tier `429` failures. Vertex AI completed the
session without a rate-limit error. The failure occurs inside the deterministic relationship
between cue resolution, stage transitions, beat eligibility, response planning, and response
validation.

## Executive Summary

The simulator can enter a state in which:

1. The counselor successfully handles the current emotional cue.
2. The cue is marked resolved and removed from the active-cue set.
3. The next progression beat is not yet available because it belongs to a later session stage.
4. When that later stage becomes available, the next beat requires acknowledgment of an active cue.
5. No active cue exists anymore, so the required counselor-response condition cannot be satisfied.
6. No beat is selected and no emotional cue is permitted in the client response plan.
7. The model naturally generates clinically plausible content in response to the counselor.
8. The validator correctly recognizes that content as a locked disclosure and rejects it.
9. A generic controlled fallback reveals no new beat and presents no new cue.
10. The same impossible state repeats on every subsequent turn.

The resulting loop is:

```text
No active cue
  -> next beat cannot satisfy its cue-response requirement
  -> no eligible or selected beat
  -> no permitted disclosure or cue
  -> generated draft crosses a locked boundary
  -> validator rejects the draft
  -> generic fallback advances nothing
  -> no active cue
```

This is primarily a state-machine and beat-authoring semantics issue. It is not primarily a model
quality problem and it is not primarily a validator problem.

## User-Visible Symptoms

The client repeatedly says:

```text
I'm still trying to put that into words. I think I need a little more time before I can say much
more about it.
```

In the reproduction session:

- 9 counselor turns completed.
- 14 generated drafts were rejected.
- 7 controlled fallbacks were shown to the student.
- Only `work_stress` and `time_management` were revealed.
- `emotional_exhaustion`, `relationship_guilt`, `identity_concerns`, and
  `loss_of_fulfillment` remained unrevealed.
- Trust reached 5 and engagement reached 4, but progression still stopped.
- The session reached the mid stage, but the first required mid-stage beat could not unlock.

The repeated fallback also contaminated the final evaluation. The evaluator reasonably interpreted
the fallback language as the client requesting processing time, then criticized the counselor for
not using silence. In reality, the fallback was generated by a disclosure-control failure rather
than an authored client reaction.

## Expected Behavior

After the counselor accurately reflects and deepens the early overwhelm cue, the system should
retain that therapeutic achievement. Once the mid-stage timing and trust conditions are met, the
client should be able to introduce the next bounded story beat, `emotional_exhaustion`.

A successful sequence should look like:

```text
work_stress revealed
  -> Stress cue handled
  -> time_management revealed
  -> Overwhelm cue handled
  -> mid-stage gate opens
  -> emotional_exhaustion becomes eligible
  -> client presents Fatigue cue and bounded exhaustion disclosure
  -> counselor acknowledges or deepens Fatigue
  -> subsequent mid-stage progression becomes available
```

## Actual Turn Sequence

### Turn 1

Runtime state:

```text
stage: early
trust: 2
engagement: 3
revealed: [work_stress]
active cue: Stress
cue response: accurately_reflected
eligible beat: time_management
selected beat: time_management
```

The generated response revealed `time_management` and presented overwhelm/frustration. This turn
worked as intended.

### Turn 2

Runtime state:

```text
stage: early
trust: 3
engagement: 3
revealed: [work_stress, time_management]
active cue: Overwhelm
cue response: deepened
eligible beat: none
selected beat: none
```

The overwhelm cue was handled successfully. Its ledger status changed to `deepened`, so it was no
longer included among unresolved active cues.

No next beat was available because `emotional_exhaustion` belongs to the mid stage while the
session was still early.

### Turn 3

Runtime state:

```text
stage: early
trust: 3
engagement: 3
active cues: []
permitted cues: []
eligible beats: []
selected beat: none
```

The counselor asked about emotional pressure. Gemini produced a plausible exhaustion response, but
the semantic validator identified `emotional_exhaustion` as a locked beat and rejected it.

### Turn 4

Runtime state:

```text
stage: mid
trust: 4
engagement: 4
active cues: []
cue response: no_active_cue
eligible beats: []
selected beat: none
```

The stage had finally advanced to mid. The thresholds for `emotional_exhaustion` were otherwise
available, but that beat was authored with:

```text
required_counselor_response: acknowledge_cue
```

Eligibility checks the current or immediately updated active-cue response. Because the previous
overwhelm cue had already been resolved, the analyzer returned `no_active_cue`. The beat remained
locked.

### Turns 5-9

The system remained in the same state:

```text
stage: mid
trust: 4-5
engagement: 4
revealed: [work_stress, time_management]
active cues: []
permitted cues: []
eligible beats: []
selected beat: none
```

Gemini repeatedly followed the counselor's increasingly deep questions and generated exhaustion,
guilt, identity, fulfillment, or sustainability content. The validator correctly detected those
locked semantic claims. The fallback introduced no recovery path.

## Why Disclosures And Cues Become Empty

The response planner derives permitted material from two sources:

1. The progression beat selected for the current response.
2. Existing unresolved emotional cues that the client may continue expressing.

During the deadlock:

```text
selected progression beat = null
unresolved emotional cues = none
```

Therefore:

```text
selected_disclosure_key = null
selected_disclosure_content = null
active_emotional_cues = []
permitted_emotional_cues = []
```

The model still sees the recent conversation and counselor message. It attempts to answer naturally,
but every meaningful continuation risks entering protected future story content.

## Why The Validator Is Not The Primary Cause

The validator detected genuine boundary violations. Examples include:

```text
Draft: "I used to love teaching... Now it just feels like another obligation."
Finding: loss_of_fulfillment explicitly_revealed, confidence 0.95
Decision: reject because loss_of_fulfillment is locked
```

```text
Draft: "I feel this constant guilt, like I'm letting people down."
Finding: relationship_guilt explicitly_revealed, confidence 1.0
Decision: reject because relationship_guilt is locked
```

Those disclosure decisions are appropriate given the response plan.

However, the validator also treats any emotional cue outside `permitted_emotional_cues` as locked.
This can reject safe, adjacent emotional language such as frustration, overwhelm, or self-doubt even
when no protected story claim is disclosed. That behavior makes the deadlock more severe, but it
does not create the initial circular dependency.

## Root Causes

### 1. Cue-response requirements are temporally ambiguous

`required_counselor_response: acknowledge_cue` currently means the response must have occurred at
eligibility-check time. It is unclear whether faculty intend it to mean:

- Acknowledge the preceding beat's cue before this beat may be presented.
- Acknowledge this beat's cue after the client presents it.
- Demonstrate cue acknowledgment at any earlier point in the session.

These meanings require different state transitions, but the authoring model represents them with
one field.

### 2. Successful cue handling is not stored as a durable beat-gate achievement

The cue ledger records that a cue was deepened, but beat eligibility relies heavily on the active or
immediately previous cue-response value. When stage timing delays the next beat, the successful
response is no longer usable as gate evidence.

### 3. Stage gating and cue gating occur on different timelines

The counselor can satisfy the clinical condition before the minimum turn count opens the next
stage. By the time the stage opens, the qualifying cue has left the active set.

### 4. The fallback has no recovery semantics

The controlled fallback is safe in terms of disclosure, but it does not:

- Present a permitted cue.
- Re-present the preceding cue.
- Select a recovery beat.
- Preserve a route to satisfying the blocked gate.

It converts a transient validation failure into a persistent progression failure.

### 5. Cue validation is stricter than disclosure validation needs to be

Protected story claims and ordinary emotional vocabulary are treated similarly. A client can safely
sound frustrated without revealing a future beat. The current validator may reject the emotion
itself even when its associated private meaning remains hidden.

### 6. Counselor presupposition is not represented in planning

The counselor can introduce a future concept directly, such as exhaustion or loss of identity. The
model follows that conversational lead, but the planner does not choose among explicit strategies:

- Confirm the concept without expanding it.
- Gently correct the counselor's assumption.
- Return to currently revealed material.
- Permit a bounded transition when other gates are satisfied.

## Solution Options

These options are not mutually exclusive.

### Option A: Persist completed cue-gate achievements

Store durable evidence such as:

```json
{
  "beat_key": "time_management",
  "cue": "Overwhelm",
  "highest_response": "deepened",
  "achieved_on_turn": 2
}
```

Later beat eligibility can ask whether a prerequisite beat's cue was acknowledged or deepened at
any valid prior turn.

Advantages:

- Removes the timing race between cue resolution and stage opening.
- Preserves clinically meaningful counselor achievements.
- Fits the current deterministic architecture.

Risks:

- Requires explicit rules about how long achievements remain valid.
- A single early acknowledgment might unlock too much later content if gates are broad.
- Must identify which preceding beat or cue satisfies each requirement.

### Option B: Split entry conditions from response expectations

Replace the ambiguous field with two concepts:

```text
entry_requirement:
  prerequisite cue handling required before beat presentation

response_expectation:
  counselor behavior expected after this beat presents its cue
```

For example:

```text
emotional_exhaustion.entry_requirement = prior_beat_deepened
emotional_exhaustion.response_expectation = acknowledge_cue
```

Advantages:

- Closely matches the real counseling sequence.
- Faculty intent becomes understandable and auditable.
- Eliminates circular requirements where a cue must be acknowledged before it exists.

Risks:

- Requires schema, builder, migration, prompt, and runtime changes.
- Existing scenarios need a compatibility conversion.

### Option C: Make cue acknowledgment a post-presentation condition

Allow a beat to become eligible from stage, trust, engagement, and prerequisites. Present its cue and
bounded disclosure first. Require acknowledgment or deepening before allowing the following beat.

Advantages:

- Simple mental model: client presents cue, counselor responds, story progresses.
- Prevents requirements on nonexistent cues.
- Natural fit for counseling interaction.

Risks:

- Changes the meaning of existing `required_counselor_response` fields.
- Some disclosures intended to require rapport before presentation need a separate entry gate.

### Option D: Add a deterministic recovery planner

When no beat and no cue are available, generate a recovery plan instead of an empty plan:

```text
re-present the most recent resolved cue at the same depth
or
respond only from already revealed facts
or
use a beat-specific guarded bridge response
```

Advantages:

- Prevents generic repeated fallbacks.
- Gives the counselor another opportunity to meet a gate.
- Can be implemented without immediately redesigning authoring fields.

Risks:

- Treats the symptom unless combined with a gate-model fix.
- Re-presenting resolved cues too often may feel repetitive.

### Option E: Separate protected disclosures from flexible emotional cues

Use two enforcement levels:

```text
Hard boundary:
  unauthorized private meanings and future semantic claims

Soft boundary:
  ordinary emotions that may be expressed without revealing protected facts
```

Advantages:

- Allows realistic adjacent affect.
- Reduces false rejection of clinically plausible responses.
- Keeps strict protection around authored private information.

Risks:

- Requires careful semantic definitions.
- Some emotional cues themselves may be intended as assessment milestones.

### Option F: Add beat-aware responses to counselor presuppositions

Detect when the counselor names an unrevealed future issue. The planner then chooses a bounded
reaction:

```text
tentative confirmation without new facts
gentle uncertainty
return to known experience
or eligible transition when thresholds permit
```

Advantages:

- Produces realistic responses when counselors lead or make assumptions.
- Reduces pressure on the validator to repair conversational mismatches.

Risks:

- Requires a semantic presupposition finding.
- Must not reward leading questions by unlocking protected content automatically.

### Option G: Use authored fallback variants per beat and gate state

Replace the single generic fallback with responses such as:

```text
Early guarded: "I know I'm stressed, but I haven't sorted out what is underneath it yet."
Cue re-presentation: "The overwhelm is still the clearest part for me right now."
Presupposition boundary: "I'm not sure I would describe it that way yet."
Processing response: "I need a moment to sit with that."
```

Advantages:

- Better realism and less evaluation contamination.
- Makes fallback behavior explicit and testable.

Risks:

- Does not resolve impossible eligibility by itself.
- More authored content to maintain.

## Candidate Combined Designs

### Minimal Stabilization

```text
Persist prior cue achievements
+ deterministic recovery planner
+ hard-disclosure/soft-cue validation split
```

This is the smallest route to eliminate the current deadlock while preserving most schemas.

### Clean Semantic Redesign

```text
Separate beat entry requirements from post-presentation response expectations
+ durable cue achievement ledger
+ explicit presupposition handling
+ beat-specific recovery behavior
```

This provides the clearest long-term authoring and runtime model, but has a larger migration and UI
cost.

### Pure Sequential Counseling Model

```text
Beat becomes eligible from prerequisite/story gates
-> client presents its cue
-> counselor response quality is assessed
-> next beat waits for required cue handling
```

This is the easiest model to explain, test, and visualize. It may need separate rapport or safety
entry gates for information that must not be introduced based only on story sequence.

## Recommended Decision Questions

Before choosing an implementation, faculty and engineering should answer:

1. Does `required_counselor_response` describe what must happen before a beat appears or after it
   appears?
2. Should a successfully handled cue remain valid as progression evidence across later turns and
   stage changes?
3. Can common emotions appear before their formal story beat if no private meaning is disclosed?
4. What should the client do when the counselor introduces a future issue prematurely?
5. Should stage advancement depend on turn count, clinical achievements, or both?
6. Should a failed draft consume any state transition, or should state commit only after an accepted
   client response?
7. How many consecutive controlled fallbacks should trigger an explicit recovery or faculty-visible
   system warning?

## Acceptance Criteria For Any Solution

A correct solution should demonstrate all of the following:

1. A successfully handled early cue can support the intended later transition after the stage opens.
2. No beat requires acknowledgment of its own cue before that cue can be presented.
3. A response plan never remains empty for multiple turns without an explicit recovery strategy.
4. Protected future semantic claims remain unavailable until their gates are satisfied.
5. Safe adjacent emotions do not cause rejection unless they reveal protected meaning.
6. Counselor presuppositions receive a bounded, realistic client response.
7. A rejected generation does not falsely mark its disclosure as revealed.
8. Repeated fallback text does not distort student evaluation as if it were authored clinical
   behavior.
9. Faculty trace shows the blocking gate, qualifying historical evidence, selected recovery strategy,
   and validator reason.
10. The full Sarah scenario can progress through all required beats with an appropriate counselor
    transcript and without a generic fallback loop.

## Required Tests

Add script-level cases for:

- Cue deepened before stage transition, then used as durable gate evidence.
- Cue ignored, preventing the next required transition.
- Cue missed and later repaired, then permitting progression.
- No active cue and no eligible beat, producing a deterministic recovery plan.
- Safe adjacent emotion without unauthorized private meaning.
- Future beat explicitly disclosed and correctly rejected.
- Counselor prematurely naming exhaustion, guilt, identity loss, or sustainability.
- Two rejected drafts followed by a beat-aware fallback.
- End-to-end Sarah transcript reaching all required beats.
- Evaluation context distinguishing authored client responses from system fallbacks.

## Current Operational Guidance

Until the architecture is corrected:

- Treat repeated controlled fallbacks as a simulator fidelity failure, not client resistance.
- Do not use affected sessions for formal student assessment.
- Review `simulation_fidelity.controlled_fallbacks` and
  `simulation_fidelity.rejected_generation_attempts` before trusting evaluation output.
- Keep the full faculty trace enabled so the original drafts and validator decisions remain auditable.
